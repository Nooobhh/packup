# packup v0.1 — Feature Brief
> 给 haze 看。spec 在 docs/ohaze/specs/2026-07-02-xhs-trip-pipeline-design.md 给 Codex 看,你不用看。
> 状态:已批准(2026-07-02 会话)

## 这是干嘛的
把「收藏夹里的小红书攻略笔记」一键转换成「按天排好、地图可视、看了就能用」的结构化行程 —— 自用版。**v0.1 重点:把小红书内容提取做透**(图文/纯图笔记都要能提取),先做好小红书,其他平台后续扩展。

## 给谁用
haze 本人(v0.1 不发布)。典型情境:计划一次真实旅行,已在小红书攒了若干篇目标城市攻略笔记,不想花 3-8 小时手动整理。

## 输入定义
| 字段 | 必填? | 说明 |
|---|---|---|
| 小红书链接 | **必填**,1-10 条 | 分享口令/短链/完整 URL 均可识别 |
| 目的地 | **必填** | 过滤与校验的锚点 |
| 天数 | 可选 | 支持固定(如 5)或浮动(如 5±1,LLM 在范围内选最优并说明);**不填则根据笔记内容量推荐天数**(行程页展示推荐理由);上限 15 天(排程质量护栏) |
| 旅行日期 | 可选 | 给了则对应到具体星期(营业时间/休息日校验更准) |
| 每日主题 | 可选,逐天 | 如 D1=city walk、D2=shopping,作为该天排程的硬约束 |
| 出行方式 | 可选 | 公共交通(默认)/自驾/步行为主 |
| 节奏 | 可选 | 紧凑/适中(默认)/慢 |

## 运行前提(v0.1)
- 全本地运行:Next.js 本地跑,内容获取走本地工具链(实测后定),**LLM 走本机 `claude -p`**(订阅内,零 API 成本;Extract 用其多模态读图,Plan 用地图工具/距离矩阵跑规划)
- 唯一外部服务依赖:地图 API(高德,个人开发者 key)
- LLM 调用点与内容获取同样做接口抽象,发布时切换为 API 计费方案,管线不动

## 排程裁决规则(冲突消解)
- 三层优先级:**客观事实(营业时间/距离/耗时)> 用户显式输入(交通/节奏/每日主题)> 笔记建议**
- 框架内具体消解由 LLM 裁量(如区分「步行是交通手段还是体验本体」),裁决框架写死在排程 prompt 中
- 所有非平凡取舍(覆盖笔记建议、内容砍削、冲突折中)必须在对应 POI 备注中写明理由,可审计

## 用户场景 (Scenarios)

### Scenario 1: 主流程 —— 链接变行程
- Given 本地起着 packup 网页,手里有 3-5 条同一目的地的小红书笔记链接
- When 粘贴链接 + 填目的地(+可选天数/浮动天数、每日主题等)→ 点「生成」
- Then 流式展示管线进度,约 3 分钟内出行程页:按天时间轴,每个 POI 含名称/地址/营业时间/推荐理由(引自笔记)/交通衔接,配当日路线地图;指定了每日主题的天,该天 POI 明显贴合主题

### Scenario 2: 重点 —— 纯图片笔记也能提取
- Given 链接中有正文寥寥数字、攻略信息全在图片里的笔记(如九宫格图片攻略)
- When 生成
- Then 图片被下载并经视觉模型提取,图中的 POI/tips 同样进入行程,与图文笔记无差别

### Scenario 3: 边界 —— 笔记内容混杂
- Given 链接里混了非目标城市或无关内容(美食探店混穿搭笔记)
- When 生成
- Then 以目的地为锚点过滤:无关 POI 不进行程,被过滤内容在结果页折叠区可见(知道丢了什么、为什么)

### Scenario 4: 失败/恢复 —— 链接取不到、POI 查不到
- Given 某条链接解析失败(风控/已删除)或某 POI 在地图服务查无此地
- When 生成继续
- Then 失败链接逐条列出原因,剩余内容照常出行程;查不到的 POI 标「未验证」进入行程(不静默丢弃、不编造坐标)

## "完成的样子" Checklist
- [ ] 能:小红书提取经真实笔记实测验证(图文、纯图两类都覆盖)——**v0.1 第一优先级**
- [ ] 能:粘 1-10 条链接 + 必填项,一次生成完整多日行程
- [ ] 能:指定每日主题时,排程遵守主题约束
- [ ] 能:天数浮动(N±n)时在范围内选最优;不填天数时按笔记内容量推荐并说明理由
- [ ] 能:偏好与笔记冲突时按三层优先级消解,取舍理由在行程中可见
- [ ] 能:行程页每天时间轴 + 当日路线地图,POI 信息完整
- [ ] 能:管线各段中间产物落盘,可单段重跑调质量
- [ ] 不能:「幻觉 POI」不加标注混进行程
- [ ] 不能:同一天地理明显折返、总时长明显超载
- [ ] 失败时能:逐条报告失败链接与原因,部分成功照常出结果

## 不做什么 (Out of Scope)
- 不发布不部署公网:无登录/支付/分享/限流/合规备案
- 不做小红书以外的平台(抖音/马蜂窝等 v0.2+;架构留 ContentFetcher 扩展位)
- 不做「目的地+偏好自动检索」输入模式
- 不做数据库(JSON 落盘)
- **不追求视觉设计**:简陋可接受,布局清晰、信息可读即达标
- 不做行程手动编辑/拖拽、导出图片/PDF

## Scope 决策
- **模式**: Selective Expansion
- **理由**: 在原范围上仅扩两处使工作流完整——纯图笔记提取(不做则大量真实笔记废掉)+ 每日主题输入(排程可控性的最小杠杆);其余一律持稳。

## Claude 替你决定的关键技术方向 (Phase 1.5 回填,事后扫一眼)
- LLM 层 = 本机 `claude -p` + `--json-schema` 结构化输出(实测 CLI 2.1.195 支持);模型默认 sonnet,env 可切 opus
- 获取层 = xhs-cli(agent-reach 底层)起步,赌「分享链接自带 xsec_token 可直读」,Spike A 实测定案;另备 ManualFetcher(本地放正文+图片)兜底,保证获取层故障不阻塞管线开发
- 排程工具接入默认走「REST 预计算距离矩阵注入 prompt」(确定性强),高德 MCP 挂 claude -p 作为升级项(Spike B 验证后切换)
- 无 DB;中间产物 `data/trips/<id>/00~40-*.json` 落盘,支持断点/单段重跑(CLI: `npm run stage`)
- 进度用原生 SSE;**不引入** Vercel AI SDK(发布切 API 时再上)
- 前端地图 = 高德 JS SDK;服务端校验 = 高德 Web 服务 REST(两个 key,.env.local)
- 测试 = vitest,LLM/网络一律 mock;真实验收靠 2 组真实笔记人工过 checklist
